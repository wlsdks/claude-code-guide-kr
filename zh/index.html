什么是 Claude Code？

就算你不会写代码，只要用平时说话的方式提需求，它就会帮你读取、修改、运行文件，是这样一款 AI 工具。

先来直观对比一下：它和你平时用的聊天工具（ChatGPT、Claude 聊天）有什么不一样。

结论聊天给的是方法 / 代码片段 —— Claude Code 直接做出能运行的成果。

关键不在于‘是否给你答案’，而在于能不能在你的项目里真正做出来、并让它跑起来。如今的聊天工具也能帮你改文字、改代码，但那只是‘在聊天里展示出来’ — 而 Claude Code 会亲手创建多个文件、把它们连接起来、运行起来，最终完成一个‘能运行的成品’。（两者用的是同一个 Claude 模型。）

AI 的工作环境 —— 你递给它什么

同样是 Claude，结果会因为你递给它什么而大不相同。工具·指示·权限·上下文 —— 把这个‘工作环境’配置得越好，它就越能自己干好。

专家们常把它叫做 ‘harness（框架）’ 或 ‘上下文工程’。听着唬人，说到底就是把环境配置好，让 AI 干得好。点点下面这五样。

Token（词元）是 AI 读文字时最小的碎片（甚至可能比一个词还小）。就像出租车计价器，文字越长跳得越高，费用和用量上限都按它来算。

AI 不是逐字读文字，而是按 token 切成小块来读。亲自输入试试。

    技能 (Skills)

    技能一句话就是‘自己的食谱’ —— 把常用的操作方法写下来，做一次以后反复用。

    来回放一下：每次从头解释，和用技能定义一次，差别有多大。

    不用技能 每次重复

    使用技能 定义一次

    技能采用只在需要时才展开的方式，不用时几乎不占上下文窗口。如果说 CLAUDE.md 是‘始终开着的备忘’，那技能就是‘需要时才翻开的手册’。

    安全使用技能

    技能·插件·MCP 也可以拿别人做的来用。方便，但 —— 一定要确认来源是否可信。

    子代理

    子代理一句话就是‘交给它干活的员工’ —— 派一项具体工作，它在自己的工作空间里做，只把结果汇报回来。

    核心很简单 —— ‘把活分出去’。

    简单类比 · 想象你是组长。大事不全自己扛，而是分给专业的组员：“这个你去调查”、“这个你来审一下”。组员各自在自己的工位干活，只汇报结果。你的工位（主对话）保持清爽。

    亲眼看看 —— 分给 6 个人 点一下回放

    ↓ 分成 6 项同时派出去

    同时进行（并行）
    6 个人一起干 —— 比一个人挨个做要快。
    主对话保持清爽
    各自在自己空间干活，只汇报‘摘要’ → 主对话不会变乱。
    各有专长
    一个人只专注一件事 → 更准确、更安全。
    • 拥有自己的工作空间（上下文窗口）。助手翻找再久，主对话也不会变乱。
    • 可以限制它能做的事。例：‘代码审查’助手只能读、不能改 —— 更安全。
    • 内置助手有 Explore（探索）、Plan（计划）、General-purpose（通用）等。其他的可以自己创建（.claude/agents/）。

    自动化 (Hook)

    Hook 一句话就是‘自动规则’ —— 在某个时刻每次都自动执行一个固定动作。

    比方就像店门口的感应铃，或手机“晚11点自动免打扰”——设好‘发生○○ → 自动做△△’。（CLAUDE.md 是‘请求’，钩子是‘规则’）

    就像‘邮件自动回复’，一旦发生某件事，它就自动运作。

    放心不用自己动手做。 配置方法稍微有点技术性，但非开发者不懂怎么做也没关系 —— 你只要对 Claude 说“做一个 Hook，每次改文件就自动整理”，Claude 就会替你做。这一页只要知道‘有这么个东西’就够了。

    工作方式 何时 → 对什么 → 怎么做

    1
    事件（何时）
    例：保存文件之后
    2
    筛选（对什么）
    例：如果它是重要的文档
    3
    动作（怎么做）
    例：自动创建一个备份

    这些场景很好用

    自动备份
    每次修改重要文件时，自动创建一个备份。
    保护
    提前防止误删重要文件。
    通知
    工作完成后发出提醒。
    • 写在配置文件（settings.json）里。可以分多个层级：所有项目通用、本项目专用、个人专用等。
    • 每次都有保证。如果说 CLAUDE.md 是‘请求’，那 Hook 就是‘规则’ —— 不会忘、每次都执行。
    • 支持数十种时机。文件变更、工具执行前后、会话开始、压缩前后等多种时刻。用 /hooks 查看。

    结论如果 CLAUDE.md 是‘请求’ —— 钩子就是‘规则’：在设定的时刻每次自动执行。

    配置稍微有点技术性，所以你对 Claude 说“做一个 Hook，每次改文件就格式化”，它就会替你做。（来源：官方文档，截至 2026 年 6 月）

    计划模式

    在直接动手改之前，Claude 会先把‘要做什么’的计划给你看，等你批准。

    在大改动或不熟悉的工作里，可以提前避免‘跑偏方向’。来回放一下流程。

    1
    切换到计划模式
    Shift+Tab 或 /plan —— 现在只读，不修改
    2
    Claude 给出计划
    “我打算这样改这 3 个文件” —— 提前展示影响和顺序
    3
    我来审阅 · 修改
    用“footer 也漏了”这样的反馈再打磨一遍
    4
    批准后执行
    “好，执行吧” → 这时才一次性修改文件

    结论不用直接改、结果跑偏 —— 先看计划再批准，大改动也安全。

    计划模式是‘只读’的，所以很安全。它是权限模式（‘权限模式’标签页）的一种。（在 Windows 的部分终端里，建议用 /plan 代替 Shift+Tab）

    权限模式

    Claude 能自作主张到什么程度，由我来定。

    比方就像决定给助手多大自主权——“每件事先问我” ↔ “常做的就自己办”。一开始多问，熟了再放开。

    简单说 · 当 Claude 要在我电脑上做点什么（改文件、执行命令）时，它要多频繁地问“这个可以做吗？” —— 这就是这项设置。一开始每次都问比较安全，熟练之后可以放开，让它自己处理常做的事。就像汽车的‘手动 ↔ 自动’。

    在 Claude Code 里用 Shift+Tab 循环切换基本模式（请求权限·接受编辑·计划）。‘自动·跳过’这类自主模式可能需要单独的选项。点一个模式，再用“查看动作”确认差异。

    ← 安全 · 确认多自主 · 确认少 →

    结论每种模式自动执行的范围不同 —— 先用‘请求权限’稳妥，熟练后再放开常用操作。

    这 5 种是菜单里能看到的模式。另外还有一个专供自动化的‘只允许预先批准的工具（dontAsk）’模式。无论哪种模式，像删除整个系统（rm -rf / · rm -rf ~）这种极端破坏都会再确认一次（‘断路器’——不会自动执行）。不满意的话可以用‘回退’（‘回退’标签页）恢复。
