What is Claude Code?

Even if you never write code yourself, just ask in plain words and it reads, edits, and runs your files for you. It's an AI tool.

Let's start by seeing how it differs from the chat you're used to (ChatGPT, the Claude chat app).

TakeawayChat shows methods / code snippets — Claude Code actually builds a working result.

The point isn't just 'does it give you an answer' — it's does it actually build it in your project and make it work. Today's chat tools can edit text and code too, but that's 'shown inside the chat.' Claude Code creates several files itself, connects them, runs them, and finishes a 'working result.' (Both use the same Claude model.)

Harness — designing the setup that lifts results

Without switching to a pricier model — give Claude a well-built “work setup” and the same model does far better. Tuning that setup is harness (context) engineering.

The five things from “AI work setup” (tools, instructions, permissions, context, extensions), fitted and tuned to the situation. Even non-developers can do it with just CLAUDE.md, @, and tidying up.

AnalogyEven the same cook — hand over prepped ingredients, write a clear recipe, keep the kitchen tidy — turns out better food. The harness is “tidying the kitchen” for the AI.

How the experts define it

Context engineering · Andrej Karpathy
“The delicate art and science of filling the context window with just the right information for the next step.” He argued this term beats “prompt engineering.”
Source · X (2025)
Agent · Simon Willison
“An agent runs tools in a loop to achieve a goal.” The harness is where you design those tools and that loop.
Source · simonwillison.net
Harness · Anthropic
The agent’s operating framework — an environment with tools, context management (compaction), progress logs, and verification loops.
Source · anthropic.com

In three layers — prompt (one instruction) ⊂ context engineering (tokens within one window) ⊂ harness (designing the environment across many windows). Each is broader than the last.

Tokens

A token is the smallest piece an AI reads text in (it can be smaller than a word). Like a taxi meter, the longer the text the higher it climbs — and your cost and usage limits are all counted in these.

    "Can it reference older content across multiple sessions?" — the question non-developers ask most

    • Sessions are independent of each other.A new session doesn't carry over the previous conversation as-is (it starts clean).
    • Instead, memory and CLAUDE.md act as a 'bridge.'They're reloaded automatically at the start of each session, so preferences and rules learned earlier carry forward.
    • To continue one specific old conversation in fulluse /resume to pick it from a list; for your most recent one, use claude --continue.
    • It's saved only on your computer.It's not auto-shared to the cloud, other devices, or coworkers. To share rules with your team, write them in CLAUDE.md.

    In short: continue one conversation as-is = /resume, carry what was learned into the next conversation = memory and CLAUDE.md. (Source: official docs, as of June 2026.)

    Context engineering — principles and budget

    The context window is a finite “budget.” Context engineering is designing what holds only what’s needed at each moment — more ≠ better.

    LLMs tend to do worse as context grows — key info gets buried (often called “context rot”). The scarce resource isn’t tokens, it’s the model’s attention.

    AnalogyPut only the relevant papers on the desk (context window) and work is fast and accurate. Pile on clutter — even on a big desk — and it gets lost.

    Four principles

    Signal vs noise
    Drop unrelated files, duplicate logs, old chat. Relevance, not volume, drives quality.
    Check just the file via @
    Context rot
    The longer it gets, the more the start and middle blur. Re-state key instructions late and explicitly.
    Check put the crux last
    Budget split
    System prompt, tool defs, CLAUDE.md, history, read files, tool output all share one window.
    Check shares on the Tokens tab
    Reclaim tools
    /compact (compress), subagents (isolate), /clear (reset) win the budget back.
    Check compress/split when long

    Bottom lineWhen results disappoint — ask what to remove to raise signal before what to add.

    What eats the context budget

    ItemRoughlyHow to cut
    System + tool defsFixed (sizable)Turn off unused MCP/tools
    CLAUDE.mdAlways-on each sessionKeep it short and current
    Chat historyGrows as it piles up/compact · /clear
    Read filesLarge share@ just the needed part
    Tool outputSearch/logs are bigNarrow scope · isolate via subagent

    Signs the context is bloated

    · Answers get slow and costly (it re-reads everything)
    · It forgets or ignores earlier instructions
    · It pulls in the wrong files or stale content
    · “Auto-compact” keeps firing

    FixWhen that happens — /clear for a new task, /compact to continue, isolate big investigations with a subagent.

    Anthropic’s four techniques

    Compaction
    Near the limit, summarize into a fresh window — keep architectural decisions and open bugs, drop redundant output. Claude Code’s /compact does this.
    Structured note-taking
    Write to memory outside the context — persistent memory at low cost. Progress files and MEMORY.md are this.
    Just-in-time retrieval
    Don’t front-load everything — keep lightweight identifiers like file paths and pull data in via tools when needed. @ is this.
    Sub-agents
    Do focused work in a clean context, then return only a distilled summary to the main agent.

    These four are the core techniques from Anthropic’s Effective context engineering — all aimed at keeping only the high-signal tokens you need in the window.

    Source · Anthropic official — code.claude.com/docs context-window·compaction (“effective context engineering,” as of Jun 2026). Context rot and signal-vs-noise are general LLM traits.

    Designing CLAUDE.md & memory

    CLAUDE.md is always-on context, auto-loaded every session. Done well it boosts consistency; done poorly it’s noise and wasted tokens.

    Principles — short, imperative, project conventions and rules, and the things it often gets wrong. Cut generic advice, long prose, and rarely-changing background.

    What goes where

    LocationWhatScope
    ~/.claude/CLAUDE.mdYour personal global prefsAll projects (you)
    ./CLAUDE.mdProject rules & conventionsTeam-shared (git)
    ./CLAUDE.local.mdYour settings for this projectThis project (you only)
    Create
    /init — analyze the project and draft one.
    auto
    Edit
    /memory — open and edit directly.
    manual
    Quick add
    # — add a one-line note mid-chat.
    inline

    Bottom lineCLAUDE.md is an “always-on rule” — shorter and fresher means better adherence. Give one-off context with @ as you go.

    A good CLAUDE.md, by example

    # Payments service ## Commands build pnpm build · test pnpm test — must pass before deploy ## Conventions Money is always integer (cents) — no floats Store dates in UTC, display only in local ## Common mistakes New APIs go under src/routes/ · auth middleware required
    Worth writing
    Build/test commands, project conventions, common traps, one line of architecture.
    Don’t write
    Generic coding wisdom, long background, frequently-changing temp notes.

    When it grows, split it out and reference other files like @docs/conventions.md. Benchmark: “skimmable in 30 seconds.”

    Hooks — rules that are enforced

    CLAUDE.md and memory are requests Claude tries to follow. A hook is a rule that always runs at a set moment — enforced regardless of what Claude decides.

    At set moments — after a file edit, before a tool runs, before a turn ends — it runs a shell command automatically. Configured in settings.json.

    AnalogyIf CLAUDE.md is a rule posted on the wall, a hook is a lock on the door — a mechanism, not a request.

    When they fire — key events

    EventWhenCommon use
    PreToolUseBefore a tool runsBlock risky commands (rm -rf)
    PostToolUseAfter edit/runAuto-format · lint
    StopBefore the turn endsRequire tests to pass first
    SessionStartSession startInject branch/issue context
    UserPromptSubmitAfter your inputPolicy checks · add context

    Config example (settings.json)

    // .claude/settings.json { "hooks": { "PostToolUse": [{ "matcher": "Edit|Write", "hooks": [{ "type": "command", "command": "pnpm run format" }] }], "Stop": [{ "hooks": [{ "type": "command", "command": "pnpm test" }] }] } }

    Bottom lineHooks = the harness’s guardrails. Nail down the “every time, always” that CLAUDE.md can’t guarantee (format, test, block).

    Exit code 2 blocks the action and feeds the reason back to Claude. type can also be http · prompt · agent, not just command. (Source: Anthropic official — code.claude.com/docs hooks, as of Jun 2026)

    Skills — injecting ability

    Write down a multi-step procedure once and it becomes an ability Claude pulls out when needed. The key — it barely costs context until used (the body loads only on use).

    Put the procedure in SKILL.md; Claude uses it when relevant, or you invoke /skill-name directly. (Custom slash commands are now merged into skills.)

    AnalogyLike shelving a thick ops manual — it takes no room day-to-day, and you open just that chapter when needed.

    Skill vs CLAUDE.md — what goes where

    CLAUDE.md
    Facts/rules that always apply. e.g. “money is integer,” build commands.
    Skill
    Multi-step procedures used occasionally. e.g. deploy, write release notes.
    When to move
    If a CLAUDE.md section grows into a procedure → pull it into a skill.

    SKILL.md structure

    # .claude/skills/release/SKILL.md --- name: release description: Write release notes, tag, deploy --- 1. Gather changes from CHANGELOG 2. Create a version tag 3. Deploy → verify # Claude reads description first, loads the body only when relevant

    Bottom lineIf you keep pasting the same procedure → make it a skill. Build it once; the team reuses it via /name.

    Lives in ~/.claude/skills/ (personal) · .claude/skills/ (project) · plugins. Follows the Agent Skills open standard (agentskills.io). (Source: Anthropic official — code.claude.com/docs skills, as of Jun 2026)

    Isolating context with subagents

    Broad exploration pollutes the main context. Delegate to a subagent and that work’s noise stays isolated — only the conclusion comes back.

    A subagent works in its own context and returns only its final message (the result) to the parent → saves the main budget and keeps it clean.

    Delegating keeps main clean

    Helpers each work in isolated context → only the conclusion returns to main.

    Good to delegate
    Broad search, multi-file investigation, long-log analysis, parallel work. Keeps main clean.
    e.g. “Find X across the codebase”
    No need to delegate
    A small single lookup or editing an already-open file. When overhead outweighs it.
    e.g. “Fix this one line”

    Bottom lineSplit big work into helpers and take only the conclusion — main context stays short, so it’s faster and more accurate.

    How to delegate

    Just ask
    “Investigate X across the codebase and report only the conclusion” → Claude spins up a helper.
    Automatically
    For big work, Claude splits it across helpers on its own.
    In parallel (fan-out)
    Run several helpers at once — broad investigation, fast.

    Cost note · helpers each spend tokens too — for a small single task, delegating actually costs more. It pays off when broad, long, or parallel.

    Source · Anthropic official — code.claude.com/docs subagents·Agent SDK (as of Jun 2026). “Returns only the final message” is the core of context isolation.

    Tool & MCP design — the interface of ability

    An agent’s ability ultimately comes from its tools (Willison: “an agent runs tools in a loop”). So designing tools well is core to the harness — the same applies when wiring external tools via MCP.

    Too many tools, vague descriptions, or verbose results — they eat context and leave Claude unsure which to use.

    AnalogyTools on a workbench — too many and you can’t find them. Only the ones you use often, clearly labeled.

    Principles of good tools

    Fewer
    Minimize the tool surface — turn off unused MCP/tools. More tools = more “which do I use?” confusion.
    Clear names & descriptions
    Describe as if to a new teammate; namespace overlaps (e.g. jira_search).
    Short results
    Paginate, filter, summarize verbose output. Concise responses can use ~1/3 the tokens.
    Consolidate
    Fold several API calls into one tool — fewer round-trips, less confusion.

    Bottom lineTools = the hands Claude actually uses. Few, clear, concise tools make the same model smarter.

    Check before adding an MCP

    · Only the servers you truly need — dozens/hundreds of tools backfire
    · Tool names that don’t collide (namespacing)
    · Results that aren’t huge per call (scope/filter)
    · Human-readable names in results, not UUIDs

    Bottom lineTools are chosen better when fewer — “more tools don’t always lead to better outcomes” (Anthropic).

    (Sources: Anthropic Writing tools for agents · Building Effective Agents · Simon Willison simonwillison.net)

    Verify & eval loop — is the harness working

    How do you know the harness works? — a loop that verifies the result and fixes the setup when it falls short. Anthropic’s harness principle: end by leaving the environment clean (tests pass, no bugs).

    Instead of checking by hand every time — make Claude self-check via tests, lint, and progress logs.

    AnalogyA factory QA station — each stage must pass to move on. Defects don’t ship even without someone watching every item.

    Automated verification devices

    Force tests/lint
    A Stop hook that “must pass to end” — ties to “Hooks.”
    Progress file
    Write state to a progress file — so even a fresh context knows how far it got (core to Anthropic’s harness).
    Plan & review
    Approve via plan mode before big changes; check with /code-review.
    Clean exit
    At the end: build/tests pass, docs tidied — “fit to leave in production.”

    Bottom lineBuild verification into the environment and Claude can guard its own quality over long runs.

    Evals — not once, but many times

    · Run the same task repeatedly to see if it succeeds consistently
    · After changing the harness, compare “better than before?” on the same bar
    · Collect failures — classify what was missing (context, tools, instructions)

    Bottom lineEvals are the harness’s compass — judge by “consistently good,” not gut feel.

    (Sources: Anthropic Effective harnesses for long-running agents · code.claude.com/docs hooks)

    Harness anti-patterns & checklist

    When results are bad, the usual cause is the harness, not the model. Common failures, their fixes, and a pre-flight checklist.

    Symptom → cause → fix

    SymptomCauseFix
    Off-target answersContext overload (noise)Drop the unrelated; @ just the file
    Random resultsVague instructionsSpecify what, how, and verification
    Slow & costlyNeglected context/compact · split sessions
    Rules ignoredStale, bloated CLAUDE.mdMake it short and current
    Tool confusionToo many tools connectedConnect only what’s needed

    Pre-flight checklist

    ✓ Is CLAUDE.md current and concise?
    ✓ Did you give only the relevant files via @?
    ✓ Did you /clear unrelated context?
    ✓ Are instructions specific (what, how, verify)?
    ✓ Are tools/MCP only what’s needed?
    ✓ Did you verify the result yourself?

    Bottom lineIf results disappoint — run through this table and checklist before a pricier model. The cause is usually the harness.

    When results are bad — check in this order

    1. Is the instruction specific? (most common cause)
    2. Did you give the relevant files via @?
    3. Is the context bloated? → /compact · /clear
    4. Is CLAUDE.md current and concise?
    5. Are tools/MCP overdone?
    6. Still? — raise the model or effort (last)

    OrderUsually it’s fixed at steps 1–3. Bumping the model is the last resort, after checking the setup.
