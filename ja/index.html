Cowork とは？

Cowork は、コードではなく 「オフィス業務」を代わりにこなしてくれる Claude です。目標を伝えるだけで、自分のファイルを直接扱って完成した成果物（Excel・文書など）を作ってくれます。

ハーネス — 結果を引き上げる「環境設計」

もっと高いモデルに替えなくても — クロードに渡す「作業環境」をしっかり整えれば、同じモデルでもずっと良くなります。この環境を調えるのがハーネス（コンテキスト）エンジニアリングです。

AI に渡す5つ — ツール・指示・権限・文脈・拡張 — を状況に合わせて揃え、調えるのが核心。5つ自体は AI 作業環境 参照。非開発者でも CLAUDE.md・@・片づけだけで十分できます。

たとえ同じ料理人でも — 食材を下ごしらえして渡し、レシピを明確に書き、厨房を整える — と良い料理になります。ハーネスは AI の「厨房を整える」仕事です。

設計は一発でなく「ループ」

一度整えて終わりではなく — 結果を見て何が足りなかったかを診断し、環境を補強して素早く繰り返します。

① 観察
結果を見る — どこでずれた？
② 診断
文脈がない？指示が曖昧？ツールがない？ノイズが多い？
③ 補強
足りないつまみを埋める — CLAUDE.md・@・ツール・整理。
④ 反復
再依頼 → また観察。調えるのは環境であってモデルでない。

自動圧縮 (auto-compact) とは?

    スキル (Skills)

    スキルはひとことで言うと「自分のレシピ」です — よく使う作業の手順を書いておき、一度作ればずっと再利用できます。

    毎回ゼロから説明するのと、スキルで一度決めておくのとの違いを再生してみましょう。

    スキルなし 毎回くり返し

    スキルあり 一度だけ定義

    スキルは必要なときだけ開く方式なので、使わないときはコンテキストウィンドウをほとんど占めません。CLAUDE.md が「いつもオンのメモ」なら、スキルは「必要なときに開くマニュアル」です。

    どんな見た目?

    # .claude/skills/週報/SKILL.md --- name: 週報 description: 週次レポートを会社の様式で作成 --- # 手順 1. 今週やったことを集める 2. 会社の様式（見出し・表）に合わせる 3. 要点のまとめを一番上に置く

    呼び出し: /週報 のように直接呼ぶか、関連する作業なら Claude が自動で適用。あらかじめ作られたスキルはプラグインとしてインストールすることもできます（「プラグイン」タブ）。

    スキルを共有する理由

    スキルは 一度作れば繰り返し再利用できる「作業指示書」です。さらに 共有するともっと良い — チームが同じやり方で働き、ノウハウが着実に積み重なります。

    例：「デプロイ」スキルに テスト → ビルド → デプロイ → 確認 の手順を入れておけば、毎回説明しなくても誰でも同じようにできます。下の利点を押してみてください。

    たとえチーム共通のチェックリスト — 毎回ドキュメントを見返して手順を抜かす代わりに、一度作っておけば新しいメンバーもそのまま従えます。

    スキルの利点 — 押してみてください

    結論スキルを共有すると — 自分だけのノウハウがチーム全体の基礎になります。みんなが同じやり方で、抜け漏れなく。

    どこまで共有されるか

    範囲どこに誰が使う
    個人~/.claude/skills/自分のすべてのプロジェクト
    プロジェクト.claude/skills/（Git で共有）このプロジェクトのメンバー
    組織管理設定会社全体
    プラグインプラグインの束インストールした誰でも

    作り方が難しければ、クロードに 「この作業をスキルにして」と頼めば代わりに作ってくれます。（出典：code.claude.com/docs — skills、2026年6月時点）

    スキルを安全に使う

    スキル・プラグイン・MCP は 他人が作ったものを受け取って使うこともできます。便利ですが — 出所が信頼できるかを必ず確認しましょう。

    スキルには 「このツールは確認なしで使っていい」（allowed-tools）といった権限が含まれることがあり、スクリプトが付いて 実行されることもあります。だからこそ信頼が重要です。

    たとえコンサルタントに仕事を任せる前 — 「どんな権限が必要ですか？ 信頼できる方ですか？ 手順を先に見てもいいですか？」と尋ねますよね。「すべて無制限のアクセスが必要だ」と言われたら 赤信号です。

    インストール前に確認すること — 押してみてください

    結論安全 = 信頼できる出所からのみ・権限は狭く・内容は一度確認。（Anthropic は外部のスキル・MCP・プラグインを一つひとつ審査しません — 信頼の判断は自分の役目。）

    いちばん安全なのは 自分で（またはクロードに頼んで）作って使うことです。外部のものを使うときは信頼できる提供者のものだけを。機密性の高い環境なら /sandbox で実行範囲を限定することもできます。（出典：code.claude.com/docs — security · skills、2026年6月時点）

    MCP (ツール連携)

    MCP = Model Context Protocol (モデルが外部とつながる標準)。ボタンを押すと連携の過程を見られます。

    かんたんにあるアプリに「Google でログイン」をつなぐと、そのアプリが自分の Google を使えますよね? MCP もそんな「連携」です — Claude に Jira・Figma のようなツールをつないであげると、Claude がそれを直接開いて仕事します。自分でいちいちコピペしなくても済むんです。

    サブエージェント

    サブエージェントはひとことで言うと「仕事を任せる社員」です — 特定の仕事を任せると自分専用の作業空間で処理し、結果だけを報告します。

    核心はシンプルです — 「仕事を分けて任せる」。

    わかりやすいたとえ · 自分がチームリーダーだと思ってください。大きな仕事を一人で全部やらず、専門のメンバーに「これは君が調べてきて」「これは君が確認して」と分けて任せますよね。メンバーはそれぞれ自分の机で働き、結果だけ報告します。自分の机（メインの会話）はきれいに保たれます。

    同時に (並列)

    メインはきれい
    それぞれ自分の空間で働き「要約」だけ報告 → メインの会話が散らかりません。
    それぞれ専門
    一人が一つのことに集中 → より正確で安全です。

    • できることを制限できます。例: 「コードレビュー」のお手伝いは読むだけで直せないように — より安全。
    • 標準のお手伝いに Explore(探索)・Plan(計画)・General-purpose(汎用) などがあります。それ以外は自分で作れます (.claude/agents/)。

    結論RPA は 決まった動作をそのまま反復（速いが画面・様式が変わると壊れやすい）、AI エージェントは 目標を理解して状況に合わせて適応します。スキルは AI に渡す 「作業指示書」です。

    誤解注意 · RPA はクロードの機能ではなく 比較のための概念です。「AI エージェント + RPA = スキル」ではなく、スキルは AI に仕事を頼む指示書にすぎません。両者は代替というより 用途が違います — 単純で変わらない大量作業には RPA が速く、変化・判断・例外処理が必要な仕事には AI エージェントが強いです。（最近は両方を組み合わせて使うこともあります。）

    なぜキャプチャを貼るの?

    言葉より速い
    画面を長々と説明しなくても、そのまま。
    正確な状態
    実際に見えているその姿を直接確認。
    デザインの通り
    デザイン案を渡せば、そのレイアウト・色で。
    エラー画面
    エラーのキャプチャを渡せば原因を早く。
    • 画像を「クリック」はできません。画面を分析するだけで、絵の中のボタンを直接押すことはしません。
    • とても小さい文字は間違えることがあります。大事な数字・文字はテキストでも一緒に書いておくと安全です。
    • 動く GIF は最初の場面だけ見ます。JPG・PNG などのふつうの画像をおすすめします。

    貼り方: キャプチャを入力欄にドラッグして置くか貼り付け (Cmd/Ctrl+V)、または画像ファイルのパスを書けば OK です。

    自動化 (フック · Hooks)

    フックはひとことで言うと「自動ルール」です — 特定の瞬間に決まった動作が毎回自動で実行されます。

    たとえお店のドアの呼び鈴や、スマホの「23 時に自動でおやすみモード」のように——「○○が起きたら → 自動で△△」を設定。（CLAUDE.md が「お願い」なら、フックは「ルール」）

    「自動返信メール」のように、あることが起きると自動で動きます。

    ご安心を自分で作る必要はありません。 設定方法は少し技術的ですが、非エンジニアは作り方を知らなくても大丈夫 — Claude に 「ファイルを直すたびに自動で整理するフックを作って」のように言葉でお願いすれば、Claude が代わりに作ってくれます。このページは「こういうものがある」とわかれば十分です。

    どう動くか いつ → 何を → どうやって

    1
    イベント (いつ)
    例: ファイルを保存した直後
    2
    フィルター (何を)
    例: それが大切な書類なら
    3
    作業 (どうやって)
    例: 自動でバックアップを1つ作る

    こんなときに便利

    自動バックアップ
    大切なファイルを直すたびに、自動でバックアップを。
    保護
    うっかり大切なファイルを消してしまうのを、あらかじめ防ぐ。
    通知
    作業が終わったら通知を送る。
    • 設定ファイル(settings.json)に書きます。全プロジェクト用・このプロジェクト用・個人用など、いくつかの段階で置けます。
    • 毎回保証されます。CLAUDE.md が「お願い」なら、フックは「ルール」 — 忘れずにいつも実行されます。
    • 数十種類のタイミングに対応します。ファイル変更・ツール実行の前後・セッション開始・圧縮の前後などさまざまな瞬間。/hooks で確認。

    結論CLAUDE.md が『お願い』なら — フックは『ルール』：決めた瞬間に毎回自動で実行されます。

    設定がやや技術的なので、Claude に「ファイルを直すたびに整形するフックを作って」のようにお願いすれば代わりに作ってくれます。（出典: 公式ドキュメント、2026年6月時点）

    プランモード

    すぐ直す前に、Claude が「何をするか」計画を先に見せて、自分の承認を受けます。

    大きな変更や慣れない作業で、「見当違いの方向」を前もって防げます。流れを再生してみてください。

    1
    プランモードに切り替え
    Shift+Tab または /plan — これで読むだけで、直しません
    2
    Claude が計画を提示
    「この 3 つのファイルをこう変えます」 — 影響と順序を前もって見せる
    3
    自分が確認 · 修正
    「footer も抜けてるね」のようなフィードバックでもう一度整える
    4
    承認したら実行
    「OK、実行して」 → そのとき一気にファイルを直します

    結論すぐ直して的外れになる代わりに — 先に計画を見て承認すれば大きな変更も安全です。

    プランモードは「読み取り専用」なので安全です。権限モード（「権限モード」タブ）の一種です。（Windows の一部のターミナルでは Shift+Tab の代わりに /plan をおすすめ）

    権限モード

    Claude がどこまで自分で進めるかを、自分で決めます。

    たとえアシスタントにどこまで任せるか——「何でも先に聞いて」 ↔ 「いつものことは自分で」。最初は確認多め、慣れたら緩める。

    かんたんに言うと · Claude が自分のコンピュータで何か（ファイルを直す・コマンドを実行する）をするとき、「これやっていいですか?」とどのくらい頻繁に聞くかを決める設定です。最初は毎回聞くのが安全で、慣れてきたらよくやる仕事は自動でやらせるように緩められます。車の「マニュアル ↔ オートマ」のようなものです。

    Claude Code では Shift+Tab で基本モード（権限リクエスト・編集の承認・プラン）を循環します。「自動・スキップ」のような自律モードは別のオプションが必要な場合があります。モードを押して「動作を見る」で違いを確認してください。

    ← 安全 · 確認が多い自律 · 確認が少ない →

    結論モードごとに自動で実行する範囲が違います — 最初は『権限リクエスト』で安全に、慣れたらよく使う操作を緩める。

    この 5 つはメニューに見えるモードです。自動化専用に「事前承認したツールだけ許可 (dontAsk)」モードも別にあります。どのモードでもシステム全体の削除 (rm -rf / · rm -rf ~) のような極端な破壊は、もう一度確認を取ります（「サーキットブレーカー」 — 自動では実行されません）。気に入らなければ「巻き戻し」（「巻き戻し」タブ）で復元できます。

    巻き戻し (チェックポイント)

    Claude は、自分が送ったリクエストごとに、その瞬間の状態を自動保存しています。気に入らなければ巻き戻せます。

    「タイムトラベル」のように、以前の時点へ行けます。点を押して巻き戻してみてください。

    状況: 「デザインをモダンに変えて」 → ファイルが 10 個変わったけど気に入りません。どう巻き戻しましょう?

    上のタイムラインの点を押したあと、復元の方法を選んでください。

    結論git を知らなくても — 気に入らなければ前の時点に一度で戻せます。（コードだけ／会話だけ／両方）

    • 開き方/rewind、または入力欄が空のときに ESC を2回 — これまで送ったリクエストの一覧が出ます。（入力欄に文字があると、2回押すと先に文字が消えます。）
    • 3 種類の復元コード+会話を一緒に / 会話だけ(コードは維持) / コードだけ(会話は維持)。
    • 限界rm·mv のようなターミナルコマンドで変わったもの、インストール(npm install)・外部の変更は巻き戻せません。

    チェックポイントは「セッション内の素早い巻き戻し」(約30日保管)、git は「永続的なバージョン管理」 — 役割が違います。長く保管したいなら git にコミットしてください。（出典: 公式ドキュメント、2026年6月時点）

    実践のコツ —— ベストプラクティス

    Anthropic チームで実証された実践習慣。多くは一つの制約から —— コンテキスト窓はすぐ埋まり、埋まると性能が落ちる。

    だから核心は —— 文脈は小さく、検証は自動で、修正は素早く。

    身につけたい習慣

    チェックを渡す
    テスト·ビルド·スクショ —— 「できたっぽい」でなく pass/fail でクロードが自分で反復。
    探索 → 計画 → 実装
    大きい·不慣れな変更は計画モードから(Shift+Tab)。小さいなら即実行。
    具体的に
    どのファイル·どの場面·「終わったら何で確認」まで書く。
    @·画像·パイプ
    @ でファイル、スクショ貼付、cat error.log | claude。
    素早く修正
    Esc で停止·Esc Esc で巻き戻し。2回外したら /clear して新プロンプト。
    /clear で文脈管理
    無関係な作業を始める時は空に —— 雑多な文脈は性能を下げる。
    調査はサブエージェントで
    ファイルを多く読む調査は隔離 —— メイン文脈をきれいに。
    CLAUDE.md は簡潔に
    「消すと失敗する？しないなら削除。」長いと逆に無視される。

    よくある失敗 —— これはダメ

    · 1セッションに雑多に混ぜる → タスクごとに /clear
    · 直し続けて蓄積 → 2回失敗で /clear + より良いプロンプト
    · CLAUDE.md 詰め込みすぎ → 容赦なく整理(またはフック)
    · 信じて確認せず → 必ず検証(できないなら出さない)
    · 無限の調査 → 範囲を絞るかサブエージェント

    要点文脈は小さく·検証は自動で·修正は素早く —— あとは実践で直感を養う。

    コンテキストはなぜ管理が必要？

    コンテキスト（文脈）は Claude の 「机」です。机が散らかると、仕事が 遅くなり、高くつき、ミスが増えます。だからこそ、こまめに片づけることが大切です。

    同じ作業を、放っておくセッションとこまめに片づけるセッションの両方に送ってみましょう。コンテキストがたまるほど、2つの結果がどう分かれるか比べてください。

    ためるだけ 放置
    机（コンテキスト）12%
    作業ごとに片づけ 管理
    机（コンテキスト）12%

    「次の作業を送る」を押して — 同じ頼みを両方のセッションに送り、結果を比べてください。

    片づけ方 — こうやって管理します

    新しい仕事の開始 = /clear
    別の作業に移るとき、机を空にします。（「セッション & フォルダ」タブ）
    長くなったら /compact
    要点だけ要約してスペースを空けます。（「圧縮」タブ）
    1 セッションに 1 つの仕事
    いくつもの仕事を混ぜません — 混ざると混乱します。
    必要なものだけ載せる
    使わないファイルや内容は、わざわざ入れません。
    • なぜ遅く・高くなるの？Claude は答えるたびに机の上すべてを読み直します。積み上がるほど毎回時間がかかり、トークン（＝費用）も増えます。
    • なぜ正確さが下がるの？コンテキストが長すぎると、途中にあった大事な内容を取りこぼしやすくなります。必要なものだけ残せば、より正確になります。
    • 放置すると自動圧縮が来ます。いっぱいになると Claude が自動で要約（自動圧縮）しますが、そのとき詳細の一部が消えることがあります。先に片づければ、自分でコントロールできます。（「コンテキストウィンドウ」「圧縮」タブ）

    ひとことで言えば — きれいな机 = 速くて正確な仕事。 作業が変わるときは /clear、長くなったら /compact だけ覚えておけば大丈夫です。

    圧縮 (compact)

    会話が長くなると「圧縮」が起こります。長く積み上がった内容が、短い要約 1 つに変わります — 机の上の書類を、要点だけ 1 枚にまとめるのと似ています。

    たとえ一日のレシートを「本日の合計」の一行にまとめて引き出しを空けるように——長い会話を短い要約に縮めて場所を空けます。

    考える深さ (effort)

    考える深さ（effort）は、Claude が答える前にどれだけ深く考えるかを決める設定です。深く考えるほど難しい問題により正確になりますが、より遅く高くつきます。

    たとえお茶を淹れるよう——さっと一煎（低）vs じっくり濃く（高・最高）。深く考えるほど正確ですが、時間と費用がかかります。

    低 · 中 · 高 · 超高 · 最高の段階があります（デフォルトは「高」）。深く考えるほど「思考（thinking）トークン」を多く使います（「トークン」タブ）。選んで違いを見てみましょう。

    難しい問題の正確さ
    応答の速さ
    トークンのコスト

    選んだぶんだけ「考える過程」が深まります 押すと長さが変わります

    結論簡単な作業は『低』（速くて安い）、難しい推論は『高/最高』（正確だが遅くて高い）。

    簡単な仕事は「低」で（速くて安い）、込み入った分析・推論は「高/最高」で。変え方: /effort または設定（「設定」タブ）。例 のバーは相対的な比較です。（出典: Anthropic 公式 — support.claude.com モデル・effort 設定）

    ultrathink · プロンプトのどこかにこの語を入れると — セッションの effort 設定を変えずにそのターンだけ深く推論。逆に「think hard」のような言い回しはクロードコードで認識されるキーワードではありません — 普通のテキストとして渡されます。（出典：code.claude.com/docs model-config、2026年6月時点）

    上級 · ultracode — /effort メニューには ultracode もあります。超高（xhigh）の推論に加えて、難しい作業では Claude が複数のヘルパーを自動的に動員（ワークフロー）して処理します。そのセッション限定のモードで、トークンを多く使うので普段は不要、とても難しい作業のときだけ選びます（Fable 5・Opus 4.8 など）。

    目標達成 (goal)

    目標（goal）は、「完了条件」だけ与えれば、その条件になるまで Claude が自分で繰り返す機能です。毎回もう一度頼む必要がありません。

    たとえ「味が決まるまで味見する」ように——条件を満たすまで自動で繰り返します。（loop は「決まった時刻ごと」）

    一度やって終わりではなく、毎ターンの後に「条件を満たしたか」を自分で判定し、満たしていなければまたやります。下で見てみましょう。

    条件の充足は別の判定で毎ターン確認し、満たされたら自動で止まります（無限ループ防止の仕組みもあります）。変え方: /goal。（出典: Anthropic 公式 — code.claude.com/docs goal）

    繰り返し実行 (loop)

    繰り返し（loop）は、決めた間隔ごとに同じ作業を自動でもう一度実行する機能です。例: 10 分ごとに「新しい注文が入ったか確認して通知」。

    たとえ「毎朝 7 時に水やり」のように——決まった間隔で自動でもう一度。（goal は「条件を満たすまで」）

    目標（goal）が「条件になるまで」なら、繰り返し（loop）は「決めた時間間隔で」です。間隔を選んでスタートしてみましょう。（シミュは時間を早送りして見せます。）

    ワークフロー (workflows) 上級

    ワークフロー（workflows）は、とても大きな仕事を Claude が自分で「作戦」を立てて、複数のお手伝い（サブエージェント）を同時に動かし、一気に処理する機能です。

    サブエージェントが「一人に任せる」なら、ワークフローは「複数人を同時に調整」します。大きなレビュー・整理・調査に強いです（Max・Team などの上位プラン）。

    例「契約書8 件を項目別に同時にレビューして」 → Claude が作戦を立てて、お手伝いを複数人いっぺんに投入

    「同時作業を見る」を押してみてください。

    コンテキストエンジニアリング — 原理と予算

    コンテキスト窓は有限の「予算」。各時点でモデルに必要な情報だけを持たせる設計がコンテキストエンジニアリング — 多い ≠ 良いです。

    LLM はコンテキストが長いほど重要な情報が埋もれ性能が落ちがち（いわゆる「context rot」）。希少なのはトークンでなくモデルの「注意」です。

    たとえ机（コンテキスト窓）に関連書類だけを置けば速く正確。大きい机でも雑物まで積めば迷子になります。

    4つの原理

    信号 vs ノイズ
    無関係なファイル・重複ログ・古い会話は外す。質を決めるのは量でなく関連性。
    点検 @ でその一つだけ
    コンテキスト劣化
    長いほど先頭・中間がぼやける。重要な指示は後で・明示的に言い直す。
    点検 要点は最後に
    予算配分
    システム・ツール定義・CLAUDE.md・履歴・読んだファイル・ツール結果が1つの窓を分け合う。
    点検「トークン」タブで比率
    回収ツール
    /compact（圧縮）・サブエージェント（隔離）・/clear（整理）で予算を取り戻す。
    点検 長くなったら圧縮・分割

    結論結果が物足りない時 — 何を足すかより何を外して信号を上げるかを先に。

    何がコンテキスト予算を食うか

    項目およそ減らし方
    システム+ツール定義固定（相当）使わない MCP・ツールを切る
    CLAUDE.md毎セッション常時短く最新に
    会話履歴溜まるほど増/compact・/clear
    読んだファイル大きな割合@ で必要な部分だけ
    ツール結果検索・ログが大範囲を絞る・サブエージェントで隔離

    コンテキスト肥大のサイン

    · 回答が遅く高コストに（毎回全部読み直す）
    · 前の指示を忘れる・無視する
    · 見当違いのファイルや古い内容を引く
    · 「自動圧縮」が頻発する

    対処その時は — 新タスクは /clear、続けるなら /compact、大きな調査はサブエージェントで隔離。

    Anthropic がまとめた4技法

    圧縮 · Compaction
    上限に近づいたら要約して新しい窓へ — アーキテクチャ決定や未解決バグは残し、冗長な出力は捨てる。クロードの /compact がこれ。
    構造的メモ · Note-taking
    コンテキスト外のメモリに記録 — 低コストで持続記憶。進捗ファイル・MEMORY.md がこの方式。
    適時ロード · Just-in-time
    先に全部入れず — ファイルパス等の軽い識別子だけ持ち、必要時にツールで取得。@ がこれ。
    サブエージェント · Sub-agents
    きれいな文脈で集中作業し、要約だけをメインに返す。

    この4つは Anthropic Effective context engineering がまとめた中核技法 — すべて窓に必要な高信号トークンだけを残すためのもの。

    出典 · Anthropic 公式 — code.claude.com/docs context-window·compaction（「effective context engineering」、2026年6月時点）。コンテキスト劣化・信号 vs ノイズは LLM 一般の特性です。

    CLAUDE.md・メモリ設計

    CLAUDE.md は毎セッション自動ロードされる常時コンテキスト。良ければ一貫性が上がり、悪ければノイズとトークンの無駄に。

    原則 — 短く・命令形・プロジェクトの慣例とルール・よく間違える点だけ。一般論・長い説明・めったに変わらない背景は外す。

    何をどこに置くか

    場所何を範囲
    ~/.claude/CLAUDE.md個人のグローバル設定全プロジェクト（自分）
    ./CLAUDE.mdプロジェクトの規則・慣例チーム共有（git）
    ./CLAUDE.local.mdこのプロジェクトの個人設定このプロジェクト（自分のみ）
    作成
    /init — プロジェクトを分析して下書き生成。
    自動
    編集
    /memory — 直接開いて編集。
    手動
    クイック追加
    # — 会話中に1行メモを追加。
    即時

    結論CLAUDE.md は「常時オンのルール」 — 短く新しいほど守られます。一回限りの文脈は @ でその都度。

    良い CLAUDE.md の例

    # 決済サービス ## コマンド ビルド pnpm build · テスト pnpm test — デプロイ前に必ず通す ## 慣例 金額は常に整数（最小単位）、浮動小数禁止 日付は UTC 保存、表示のみローカル ## よく間違える点 新 API は src/routes/ 配下 · 認証ミドルウェア必須
    書く価値あり
    ビルド/テストコマンド、プロジェクト慣例、よく踏む罠、一行アーキテクチャ。
    書かない
    一般的なコーディング常識、長い背景、頻繁に変わる一時メモ。

    長くなったら分割し、@docs/conventions.md のように他ファイルを参照。基準は「30秒で読める量」。

    2つの記憶 — 自分が書く vs クロードが書く

    CLAUDE.md自動メモリ
    誰が書く自分クロード（修正・好みを見て）
    内容指示・ルール学んだこと・パターン
    場所./CLAUDE.md 等~/.claude/projects/.../memory/
    読込毎セッション全文毎セッション MEMORY.md 先頭200行（残りは必要時）
    用途コーディング標準・ワークフロー・設計ビルドコマンド・デバッグ知見・発見した好み

    結論どちらも「文脈」にすぎず — 強制ではない。必須はフックで（「フック」参照）。自動メモリは /memory で確認・編集。

    アンチパターン · 200行超・矛盾するルール・古い情報 → かえって守られなくなる。「30秒で読める量」に収めて。（出典：code.claude.com/docs — memory·settings、2026年6月時点）

    フック — 強制されるルール

    CLAUDE.md・メモリはクロードが読んで守ろうとするお願いです。フック（hook）は決まった瞬間に必ず実行されるルール — クロードの判断と無関係に強制されます。

    ファイル編集後・ツール実行前・ターン終了前といったタイミングでシェルコマンドを自動実行。settings.json に設定します。

    たとえCLAUDE.md が壁に貼った心得なら、フックはドアの鍵 — お願いでなく装置です。

    いつ発火するか — 主なイベント

    イベントタイミングよくある用途
    PreToolUseツール実行の直前危険コマンド遮断（rm -rf 等）
    PostToolUse編集・実行の直後自動整形・lint
    Stopターン終了の直前テスト通過後に終了
    SessionStartセッション開始ブランチ・issue 文脈を注入
    UserPromptSubmit入力の直後ポリシー検査・文脈追加

    設定例 (settings.json)

    // .claude/settings.json { "hooks": { "PostToolUse": [{ "matcher": "Edit|Write", "hooks": [{ "type": "command", "command": "pnpm run format" }] }], "Stop": [{ "hooks": [{ "type": "command", "command": "pnpm test" }] }] } }

    結論フック = ハーネスのガードレール。CLAUDE.md では保証できない「毎回必ず」をフックで固定（整形・テスト・遮断）。

    終了コード 2 でその動作を止め、理由がクロードに伝わります。type は command の他に http・prompt・agent も。（出典：Anthropic 公式 — code.claude.com/docs hooks、2026年6月時点）

    スキル — 能力の注入

    よく使う複数手順を一度書けば、クロードが必要時に取り出す能力になります。要は — 使わない時はほぼ文脈を食わない（使う時だけ本文が読み込まれる）。

    SKILL.md に手順を書き、関連するとクロードが自動で使う、または /スキル名 で直接呼び出します。（カスタムスラッシュコマンドも今はスキルに統合。）

    たとえ分厚い業務マニュアルを本棚に置くようなもの — 普段は場所を取らず、必要な時だけその章を開きます。

    スキル vs CLAUDE.md — どこに何を

    CLAUDE.md
    常に適用する事実・ルール。例：「金額は整数」、ビルドコマンド。
    スキル
    たまに使う複数手順。例：デプロイ、リリースノート作成。
    移す基準
    CLAUDE.md の一節が「手順」に育ったら → スキルへ。

    SKILL.md の構造

    # .claude/skills/release/SKILL.md --- name: release description: リリースノート作成・タグ・デプロイ --- 1. CHANGELOG から変更点を集める 2. バージョンタグを作る 3. デプロイ → 確認 # まず description だけ読み、関連時のみ本文（手順）を読み込む

    結論毎回同じ手順を貼るなら → スキルに。一度作れば、チームが /名前 で再利用。

    場所は ~/.claude/skills/（個人）・.claude/skills/（プロジェクト）・プラグイン。Agent Skills オープン標準（agentskills.io）に準拠。（出典：Anthropic 公式 — code.claude.com/docs skills、2026年6月時点）

    サブエージェントで文脈を隔離

    広い探索はメイン文脈を汚染します。サブエージェントに委譲すれば、その作業のノイズは隔離され、結論だけが戻ります。

    サブエージェントは独立した文脈で働き、最終メッセージ（結果）だけを親に返します → メイン予算を節約し、きれいに保ちます。

    委譲でメインがきれいに保たれる

    補助はそれぞれ隔離された文脈で働き → 結論だけメインに戻ります。

    委譲が向く
    広い検索・複数ファイル調査・長いログ分析・並列作業。メインをきれいに保つ。
    例「コードベース全体から X を探して」
    委譲は不要
    小さな単発の確認・既に開いたファイルの修正。オーバーヘッドが上回る時。
    例「この一行だけ直して」

    結論大きな作業は補助に分け、結論だけ受け取る — メイン文脈が長くならず、速く正確に。

    どう委譲するか

    言葉で頼む
    「コードベースで X を調査して結論だけ」→ クロードが補助を立てて処理。
    自動で
    大きな作業ならクロードが自動で補助に分けます。
    並列で（fan-out）
    複数の補助を同時に — 広い調査を速く。

    コスト注意 · 補助もそれぞれトークンを使います — 小さな単発では委譲がかえって高い。広い・長い・並列で得。

    出典 · Anthropic 公式 — code.claude.com/docs subagents·Agent SDK（2026年6月時点）。「最終メッセージだけ返す」が文脈隔離の核心です。

    ツール・MCP 設計 — 能力のインターフェース

    エージェントの能力は結局ツール（Willison：「エージェントはツールをループで回す」）。だからツールをうまく設計するのがハーネスの核心 — MCP で外部ツールを繋ぐ時も同じ。

    ツールが多すぎ・説明が曖昧・結果が冗長だと — 文脈を食い、クロードがどれを使うか迷います。

    たとえ作業台の工具 — 多すぎると見つからない。よく使う物だけ、ラベルを明確に。

    良いツールの原則

    少なく
    ツール面を最小化 — 使わない MCP・ツールは切る。多いほど「どれを？」と迷う。
    明確な名前・説明
    新しい同僚に説明するように。重なるなら名前空間（例 jira_search）。
    結果は短く
    冗長な出力はページング・フィルタ・要約。簡潔なら同じ情報を約 1/3 のトークンで。
    機能を統合
    複数の API を一つのツールに — 往復も迷いも減る。

    結論ツール = クロードが実際に使う手。少なく・明確に・簡潔なツールが同じモデルを賢くする。

    MCP を繋ぐ前に点検

    · 本当に必要なサーバだけ — 数十・数百のツールは逆効果
    · ツール名が衝突しない（名前空間）
    · 一度の結果が大きすぎない（範囲・フィルタ）
    · 結果は人が読める名前で（UUID でなく）

    結論ツールは少ないほどうまく選ぶ — 「ツールが多いほど良い結果とは限らない」（Anthropic）。

    （出典：Anthropic ツール設計ガイド · Building Effective Agents · Simon Willison simonwillison.net）

    検証・評価ループ — ハーネスがうまく回るか

    ハーネスがうまく動くかどう分かる？ — 結果を検証し、足りなければ環境を直すループ。Anthropic のハーネス原則：終了時に「環境をきれいに」（テスト通過・バグなし）。

    毎回人が確認する代わりに — テスト・lint・進捗記録でクロードに自己点検させる。

    たとえ工場の品質検査台 — 各段階を通らないと次へ進めない。誰も見ていなくても不良は出ない。

    自動検証の仕掛け

    テスト/lint を強制
    Stop フックで「通らないと終了不可」 — 「フック」と連携。
    進捗ファイル
    状態を進捗ファイルに書く — 新しい文脈でもどこまで進んだか分かる（Anthropic ハーネスの核心）。
    計画・レビュー
    大きな変更前は計画モードで承認、/code-review で点検。
    きれいに終える
    終了時にビルド・テスト通過・文書整理 — 「本番に置ける状態」。

    結論検証を環境に仕込めば、クロードは自分で品質を守りながら長く働ける。

    評価（eval）— 一度でなく何度も

    · 同じ作業を何度も回し、一貫して成功するか見る
    · ハーネスを直した後、同じ基準で「前より良いか」を比較
    · 失敗事例を集め — 何が足りなかったか（文脈・ツール・指示）を分類

    結論評価はハーネス改善の羅針盤 — 勘でなく「一貫して良いか」で判断。

    （出典：Anthropic Effective harnesses for long-running agents · code.claude.com/docs hooks）

    ハーネスのアンチパターンと点検表

    結果が悪い時、よくある原因はモデルでなくハーネスです。よくある失敗とその対処、そしてリリース前チェック。

    症状 → 原因 → 対処

    症状原因対処
    的外れな回答文脈の過積載（ノイズ）無関係を外し @ でその一つだけ
    バラバラな結果曖昧な指示何を・どう・検証まで具体化
    遅く高コスト文脈の放置/compact・セッション分割
    ルール無視古く肥大した CLAUDE.md短く・最新に
    ツール混乱ツールの繋ぎすぎ必要なものだけ接続

    リリース前チェック

    ✓ CLAUDE.md は最新・簡潔か？
    ✓ 関連ファイルだけ @ で渡したか？
    ✓ 無関係な文脈は /clear したか？
    ✓ 指示は具体的か（何を・どう・検証）？
    ✓ ツール/MCP は必要なものだけか？
    ✓ 結果を自分で検証したか？

    結論結果が物足りなければ — 高いモデルの前にこの表とチェックを一巡。原因はたいていハーネスです。

    結果が悪い時 — この順で点検

    1. 指示は具体的か？（最も多い原因）
    2. 関連ファイルを @ で渡したか？
    3. 文脈は肥大していないか？→ /compact·/clear
    4. CLAUDE.md は最新・簡潔か？
    5. ツール/MCP が過剰でないか？
    6. それでも — モデルや effort を上げる（最後）

    順番多くは1〜3で解決します。モデルを上げるのは環境を点検した後の最終手段。

    出典 · Anthropic 公式 — code.claude.com/docs（context-window·memory·subagents）。症状-原因は現場でよくあるパターンの整理です（2026年6月時点）。

    ハーネス一覧 — つまみの地図

    進んだ内容すべてを一枚に。モデルはそのままに環境のつまみを設計 — 各項目を押すとその章へ。

    結果が物足りない時は上からつまみを点検。高いモデルは一番最後。

    つまみ一覧

    つまみ何強制?詳細
    文脈何を入れ何を出すか — 予算・信号対ノイズお願いコンテキストエンジニアリング

    三層で見ると

    ハーネス — 複数の窓をまたぐ環境設計
    コンテキストエンジニアリング — 一つの窓内のトークン管理
    プロンプト — 一度の指示

    上ほど広い。プロンプトだけでも良いが、長く複雑なほど外側の二層（文脈・ハーネス）が結果を決めます。

    結論ハーネスエンジニアリング = モデルでなくこのつまみの設計。結果が物足りなければ表を上から点検。

    失敗チェックリストは アンチパターン & チェックリスト の章に。

    ヘッドレス·CI — claude -p

    ターミナル対話なしで — 一行 claude -p "..." がクロードをスクリプト·CIで実行。ツールとエージェントループはそのまま。

    入力をパイプで渡し結果を受け取る（JSON も）。PR レビュー·リリースノート·lint などの自動化に。

    たとえGUI アプリをコマンドラインで動かす — 人が打たなくてもパイプラインがクロードを呼ぶ。

    よく使うフラグ

    フラグ何
    -p "..."非対話実行（print）
    --allowedTools "Read,Edit,Bash"ツールを事前承認 — プロンプトなし
    --output-format json構造化結果（.result·コスト含む）
    --permission-modeacceptEdits（自動編集）·dontAsk（ロックCI）
    --bareローカル設定を無視 — どのマシンでも同じ結果（CI 推奨）

    実例

    # PR diff をセキュリティ観点でレビュー（JSON 出力） gh pr diff "$1" | claude -p \ --append-system-prompt "セキュリティエンジニアとして脆弱性をレビュー" \ --output-format json # ビルドエラーの原因分析 → ファイルへ cat build-error.txt | claude -p "このビルドエラーの根本原因を簡潔に" > out.txt

    結論クロードをツールのように — git·gh·CI とパイプで繋いで自動化。GitHub Actions·GitLab CI も対応。

    （出典：code.claude.com/docs headless · GitHub Actions、2026年6月時点）

    権限·セキュリティ — サンドボックス & deny

    自律実行ほど安全境界が重要。サンドボックスは Bash のファイル·ネットワークアクセスを OS が隔離し、deny ルールは危険な動作を元から遮断。

    /sandbox で有効化（macOS Seatbelt·Linux bubblewrap）。CLAUDE.md（お願い）と違い — モデルの判断と無関係に強制されます。

    たとえ工場の安全フェンス — 「気をつけて」のお願いでなく物理的な境界です。

    三重の防御

    deny ルール
    permissions.deny：Bash(rm *)·機微パス — 常に遮断。
    サンドボックス
    OS が遮断：作業フォルダ外への書込·未許可ドメインへのアクセス。
    managed 強制
    組織が failIfUnavailable·allowUnsandboxedCommands: false で開発者に緩めさせない。
    資格情報保護
    sandbox.credentials で ~/.aws·トークンをサンドボックスから隠す。

    設定例 (settings.json)

    // .claude/settings.json { "sandbox": { "enabled": true, "filesystem": { "denyRead": ["~/"], "allowRead": ["."] } }, "permissions": { "deny": ["Bash(rm -rf *)"] } }

    結論自律と安全は両立 — deny+サンドボックスで境界を固め、その中でクロードが自由に働く。

    （出典：code.claude.com/docs sandboxing · security、2026年6月時点）

    エージェント SDK — クロードコードをライブラリに

    クロードコードを動かす同じツール·エージェントループ·コンテキスト管理を — Python·TypeScript で直接プログラミング。

    query() 一回でクロードがファイル読み·コマンド実行·編集を自分で。日常開発は CLI、CI·本番自動化は SDK。

    たとえCLI が完成アプリなら、SDK はそのエンジンを自分のコードに差し込むもの。

    使い方 (Python · TypeScript)

    # Python — pip install claude-agent-sdk from claude_agent_sdk import query, ClaudeAgentOptions async for m in query( prompt="auth.py のバグを見つけて直して", options=ClaudeAgentOptions(allowed_tools=["Read","Edit","Bash"])): print(m) // TypeScript — npm i @anthropic-ai/claude-agent-sdk import { query } from "@anthropic-ai/claude-agent-sdk"; for await (const m of query({ prompt: "auth.ts のバグを見つけて直して", options:{ allowedTools:["Read","Edit","Bash"] } })) console.log(m);

    どれをいつ

    CLI
    日常開発·単発·対話 — ターミナルで直接。
    SDK
    CI·カスタムアプリ·本番 — 自分のプロセスで query() 実行。
    Managed Agents
    Anthropic がサンドボックス·セッションを運用する REST API — インフラ不要で本番。

    結論能力は同じ、インターフェースが違うだけ — 多くのチームが CLI(開発)+SDK(本番) を併用。

    pip install claude-agent-sdk · npm i @anthropic-ai/claude-agent-sdk · API キー必要(ANTHROPIC_API_KEY)。（出典：code.claude.com/docs agent-sdk、2026年6月時点）

    コスト·観測 — トークンを管理する

    クロードコードはトークン課金（エンタープライズ平均開発者あたり1日約 $13）。コンテキストが大きいほど高い — 小さく保つのが肝心。

    /usage で使用量を確認、チームはワークスペースに支出上限を設定。プロンプトキャッシュ·自動圧縮は既定で適用。

    たとえタクシーのメーター — 遠く（コンテキスト大）ほど料金↑。荷物を減らせば安い。

    コスト削減の要点

    モデル選択
    多くは Sonnet、複雑な時だけ Opus。サブエージェントは haiku。
    整理
    /clear·/compact で不要な文脈を消す。
    MCP を絞る
    使わないサーバを切る — 可能なら CLI ツール(gh·aws)が軽い。
    CLAUDE.md→スキル
    常時ロードを、必要時だけ読み込むスキルへ（200行以内に）。
    委譲·具体的に
    冗長な作業はサブエージェントへ；依頼は具体的に（範囲を狭く）。

    チームのコスト·観測

    · /usage でスキル·サブエージェント·MCP 別の割合を確認
    · API はワークスペースに支出上限·レート制限
    · エージェントチームは約7倍のトークン — 小さく·短く·終わったら終了
    · OpenTelemetry·analytics で組織レベルの観測

    結論コンテキストを小さく保てば — 品質は上がりコストは下がる（同じ方向）。

    （出典：code.claude.com/docs costs、2026年6月時点）

    モノレポ·大規模コードベース

    数百万行·多パッケージのリポジトリでは、既定が無関係なファイルで文脈を埋めます。作業領域だけに絞るのが肝心。

    ディレクトリ別 CLAUDE.md、無関係ファイルの除外、生成物のブロックで文脈を小さく保ちます。

    たとえ大きな図書館で必要な棚だけ案内してもらう — 全体を見て回らない。

    絞り方

    ディレクトリ別 CLAUDE.md
    パッケージごとにルール — ルート(共通)+下位(その領域だけ)を読込。
    無関係を除外
    claudeMdExcludes で触らないパッケージの CLAUDE.md をスキップ。
    生成物をブロック
    permissions.deny：Read(./**/dist/**)·vendor — 読ませない。
    コードインテリジェンス
    LSP プラグインで定義·参照へジャンプ — ファイル走査の代わりに。
    開始位置
    そのパッケージで claude 起動 → その下位だけ見える。

    設定例（パッケージ別 settings.json）

    // packages/api/.claude/settings.json { "permissions": { "deny": ["Read(./**/dist/**)", "Read(./vendor/**)"] }, "worktree": { "sparsePaths": [".claude", "packages/api", "packages/shared"] } }

    結論肝心は「作業している部分だけ」 — 見える量が少ないほど速く·正確で·安い。

    （出典：code.claude.com/docs large-codebases、2026年6月時点）

    プラグイン制作·共有 — チーム標準をパッケージに

    これまでのスキル·コマンド·フック·MCP を一つの束にして — マーケットプレイスでチームやコミュニティに配布。一度のインストールで標準が広がる。

    プラグイン = 複数拡張の束。マーケットプレイス = プラグイン一覧を持つ git リポジトリ。

    たとえアプリストア — 誰かの束を検索·インストールし、更新も自動。

    作って配布する

    1. プラグインを作る — スキル·コマンド·フック·MCP·LSP を束ねる。
    2. marketplace.json にプラグイン一覧·場所を定義。
    3. git ホスト(GitHub·GitLab)にプッシュ。
    4. ユーザーが /plugin marketplace add → インストール。update で更新。

    マーケットプレイス例

    // .claude-plugin/marketplace.json { "name": "team-tools", "plugins": [ { "name": "deploy", "source": "./plugins/deploy" } ] }

    結論一人のノウハウ(スキル·フック)をパッケージ化 — チーム全員が同じ標準を一度に。

    （出典：code.claude.com/docs plugin-marketplaces · plugins、2026年6月時点）

    エージェントチーム — 複数のクロードが同時に

    大きな仕事を複数のクロード（チームメイト）が同時に — 共有タスクリストを持ち、互いにメッセージして協働。レビュー·調査·並列開発に強い。

    リード(メインセッション)がチームメイトを起こし調整。サブエージェントと違い — メイト同士が直接会話し、自分でタスクを取ります。（実験的）

    たとえサブエージェントが「使い走り」なら、エージェントチームは議論し合う同僚。

    サブエージェント vs チーム

    観点サブエージェントエージェントチーム
    文脈各自、結果だけ回収各自、完全独立
    連絡メインへ報告のみメイト同士で直接
    調整メインが管理共有タスクリスト·自己調整
    コスト低高(~7倍)
    向き結果だけ要る集中作業議論·協働が要る複雑作業

    いつ·どう使う

    · 有効化：CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1（実験·既定オフ）
    · 向く：並列レビュー(セキュリティ·性能·テスト各)·対立仮説デバッグ·横断機能
    · 推奨：3〜5名で開始、ファイル衝突しないよう分担
    · 並列セッションを自分で回すなら — git ワークツリーで隔離

    結論並列が本当に得な時だけ — 議論·独立して進められる仕事に。そうでなければサブエージェントが安く十分。

    （出典：code.claude.com/docs agent-teams · worktrees、2026年6月時点）

    プロンプトキャッシュ — なぜ頻繁に変えないか

    クロードは毎ターン文脈全体を再送します。プロンプトキャッシュは変わらない前半を再利用し速く安い（キャッシュ読取は入力の約10%）。自動適用。

    先頭から正確に一致する部分だけキャッシュ — だから前が変わると後ろが全部無効に。

    たとえ本を毎回1ページ目から読まず — 読んだ所まで覚え、新しいページだけ読む。

    三層構造（上ほど変わりにくい）

    層内容いつ変わる
    システムプロンプトツール定義·出力スタイルアップグレード時
    プロジェクト文脈CLAUDE.md·メモリセッション開始·/clear·/compact
    会話メッセージ·ツール結果毎ターン

    キャッシュを壊す vs 保つ

    壊す(遅くなる)
    モデル変更·/effort·MCP 接続/切断·圧縮·アップグレード。
    保つ
    ファイル編集·スキル呼出·計画モード·/rewind·サブエージェント。

    ヒントセッション開始でモデル·effort を決め、/compact はタスクの合間に — キャッシュ命中率↑·コスト↓。

    （出典：code.claude.com/docs prompt-caching、2026年6月時点）

    高度なワークフロー — インタビュー·並列·CLI

    前のコツに続き — 大きな機能と品質チェックに特に効くパターン。

    3つの要点：始める前にクロードにインタビューさせる、実装とレビューを別セッションに、外部サービスは CLI ツールで。

    たとえ設計図を先に描き(インタビュー)、施工者と検査官を分ける(並列レビュー)ようなもの。

    クロードにインタビューさせる

    私：「予約システムを作りたい。AskUserQuestion で詳しくインタビューして — 技術実装·UI/UX·エッジケース·トレードオフまで。終わったら SPEC.md に仕様を書いて。」 → クロードが質問 → 回答 → SPEC.md 完成 → 新セッションでその仕様から実装

    Writer / Reviewer 並列

    セッションA(執筆)セッションB(レビュー)
    「API のレートリミッターを実装して」
    「@rateLimiter.ts をレビューして — エッジケース·競合·既存パターンとの一貫性を確認」
    「このフィードバックを反映して：[…]」

    CLI ツールを使わせる

    gh
    issue·PR 作成/閲覧 — クロードが直接使用(未導入だと API レート制限に)。
    aws·gcloud·sentry-cli
    外部サービス連携で MCP より文脈効率的。
    不慣れな CLI も可
    「foo-cli --help で学んでから A·B·C を解決して」— クロードが自習。

    要点新セッションのきれいな文脈こそ本当の力 — 書いたばかりのコードに引きずられないレビュー、実装前に練られた仕様。

    （出典：Anthropic 公式 code.claude.com/docs best-practices、2026年6月時点）

    IDE 統合 — VS Code

    CLI の代わりに VS Code 拡張機能でクロードコードを使う — 並列 diff、選択の自動認識、GUI でのプラグイン管理。

    Extensions で「Claude Code」を検索してインストール。エディタ右上の ✱(Spark)アイコンで開く。

    たとえCLI が数字の計器なら、拡張機能はグラフィカルなダッシュボード — 同じエンジン、違う窓。

    知っておくと便利

    選択範囲を自動認識
    コードをドラッグ選択すればクロードが自動で見る。Option+K(Mac)/Alt+K で @file.ts#5-10 参照を挿入。
    並列 diff レビュー
    変更前後を比較して承認。diff 内で直接直しても認識される。
    @ファイル·@フォルダ·@ターミナル
    @auth(あいまい一致)·@src/(フォルダ)·@terminal:名前(ターミナル出力参照)。
    プラグインも GUI で
    /plugins でインストール·マーケットプレイス管理 — CLI と設定共有。
    @browser で Chrome 自動化
    Claude in Chrome 拡張と連携 — @browser で Web アプリのテスト·コンソール確認。
    CLI とセッション共有
    同じ会話履歴 — claude --resume で CLI から続けられる。

    よく使うショートカット

    ショートカット動作
    Cmd/Ctrl+Escエディタ↔クロードのフォーカス切替
    Option/Alt+K選択範囲の @メンション挿入
    Cmd/Ctrl+Shift+Esc新しい会話をタブで開く
    Shift+Enter改行(送信しない)

    要点拡張機能と CLI は同じエンジン、違うインターフェース — 一部機能(!、タブ補完)は CLI 専用なので、統合ターミナルの claude も併用。

    （出典：code.claude.com/docs vs-code、2026年6月時点）

    大規模移行 — ファンアウト·自動モード

    「React を全部 Vue に」のようなチーム規模の大仕事 — 一つずつ会話でやらず、スクリプトがクロードを繰り返し呼ぶようにする。

    パターン：一覧生成 → スクリプトループ → 少数で検証 → 全体実行。人手なしで安全に回すにはauto モードを足す。

    たとえ一人が2,000回繰り返す代わりに — 同じ指示をコンベアベルトに流す。

    ファンアウト — ファイルごとに呼び出す

    # 1) 対象一覧を作る 「2000ファイルのうち移行が必要な物を全部リストして」→ files.txt # 2) スクリプトで一つずつ claude -p 呼び出し for file in $(cat files.txt); do claude -p "$file を React から Vue に移行。OK か FAIL のみ答えて" \ --allowedTools "Edit,Bash(git commit *)" done # 3) 数件で先に検証してから全体実行

    自動モード — 無人実行

    人が介在せず
    claude --permission-mode auto -p "lint エラーを全部直して" — 分類器が危険な物だけ止め、他は通す。
    範囲を絞る
    --allowedTools で必要な物だけ — 無人であるほど重要。
    まず少数で検証
    2〜3ファイルでプロンプトを練ってから全体実行 — 最初から2,000個回さない。
    デバッグ時は verbose、本番では静かに
    --verbose は開発中だけ、本番バッチでは切ってログをきれいに。

    要点大きな仕事ほど — 会話でなくスクリプトがクロードを動かす。チーム全体が同じパターンで繰り返し作業を自動化できる。

    （出典：Anthropic 公式 code.claude.com/docs best-practices · permission-modes、2026年6月時点）

    AI コード品質ゲート — 敵対的レビュー

    クロードが無人で長く働くほど — 独立した検証が重要。書いたばかりのコードに引きずられない新しい文脈が diff だけ見て再点検。

    実装したセッション自身に採点させない — サブエージェントを新しく立て「計画通りになったか」だけ確認させる。

    たとえ採点を出題者本人でなく別の先生がするようなもの — 偏りがない。

    レビュープロンプト例

    use a subagent to review the rate limiter diff against PLAN.md. Check that every requirement is implemented, the listed edge cases have tests, and nothing outside the task's scope changed. Report gaps, not style preferences.

    品質ゲートの原則

    結果だけ見せる
    レビュアーにはdiff と基準(PLAN.md)だけ — 実装過程の言い訳は渡さない。
    足りない物だけ聞く
    「要件は全部済んだか·記載のエッジケースにテストがあるか·範囲外の変更はないか」— スタイルの好みは除く。
    /code-review で手早く
    バンドルされた /code-review スキルが新しいサブエージェントで diff を見て即座に結果を返す。
    過剰修正に注意
    レビュアーは大抵何か見つける(それが仕事だから) — 正確性·要件関連だけ採用し、他は無視。

    要点無人実行が長引くほど — 別の目(新しい文脈)でもう一度見るのがチームの安全網。

    （出典：Anthropic 公式 code.claude.com/docs best-practices、2026年6月時点）

    チームプレイブック — 導入チェックリスト

    見てきた実践内容を、チームが実際に導入する順番で整理。チームリードがそのまま展開チェックリストとして使えます。

    2層に分けます — 一度だけ設定する物と、毎回の作業で使うパターン。

    たとえ新人に机·鍵·マニュアルを一度揃え(設定)、その後は業務マニュアル通りに働く(パターン)ようなもの。

    ① チーム設定 — 一度だけ

    項目なぜ詳細
    CLAUDE.md 共有チーム全員が同じルール·慣例に従うためCLAUDE.md
    スキルライブラリ繰り返し手順を一度作りチームが再利用スキル、なぜ共有するか
    プラグイン配布スキル·フック·MCP を束ね一度の導入で標準化プラグイン制作
    IDE 拡張各メンバーが VS Code に導入 — diff·プラグイン GUIIDE 統合 (VS Code)

    ② 仕事の規模別パターン — 毎回

    規模パターン詳細
    小さな仕事直接頼む — 具体的に、検証手段付きで実践のコツ
    大きな機能インタビュー→SPEC.md→新セッションで実装、Writer/Reviewer 並列高度なワークフロー
    大規模移行ファンアウト(スクリプト繰り返し呼び出し)+ auto モード大規模移行の自動化
    品質確認新しい文脈が diff だけを見る敵対的レビューAI コード品質ゲート

    結論設定は一度、パターンは毎回 — この2つの表がチーム全体のクロードコード活用法を凝縮しています。

    新しいメンバーが来たら — ①4つの設定を揃え、②仕事の規模に合うパターンを伝える。それが全てのオンボーディングです。
